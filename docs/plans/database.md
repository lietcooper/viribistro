# Database Plan — Prisma schema, migrations, seed

## Context

The Intelligent Bistro stores users, menu items, orders, and full AI conversation history in PostgreSQL via Prisma. The schema is fully specified in `CLAUDE.md` lines 46–66; this plan turns that into a TDD-ordered build sequence. The database is the foundation for backend, auth, and AI-agent work — it must be done first.

The menu seed is also part of this module because the AI agent's system prompt needs a real, populated menu to reason about during early development.

---

## Tech stack & key decisions

- **PostgreSQL** (Railway managed Postgres in prod, local Postgres or Docker for dev).
- **Prisma ORM** — single source of truth for schema + migrations + typed client.
- **TypeScript** throughout (`backend/src/**/*.ts`).
- `tags` on `MenuItem` as `String[]` (Postgres native text array) — simple `has`/`hasSome` queries are enough for the All/Starters/Mains/Desserts/Drinks + tag filters.
- Enums in Prisma:
  - `Provider { google, local }`
  - `OrderStatus { pending, confirmed }`
  - `MessageRole { user, assistant, tool }`
  - `Category { starters, mains, desserts, drinks }` (also OK as a `String` if we want recruiter-curated categories later; default to enum for safety).
- `MenuItem.price` as `Decimal(10,2)`. Tradeoff: `Decimal` requires `.toString()` or `.toNumber()` at the API boundary, but it avoids the integer-cents conversion bugs that plague restaurant apps. Document this in the route handlers.
- `Conversation.sessionId` is a `String` (UUID generated client-side or by the server) and is `@unique`. `userId` is nullable so anonymous sessions can still chat (per `CLAUDE.md` "Cart state: keep cart in memory" — chat shouldn't require auth either, though orders do).
- Cascade deletes: when a `Conversation` is deleted, its `Message[]` go with it; same for `Order` → `OrderItem[]`.

## TDD task list

Each numbered step is a `Test → Implement → Commit` block. Do not advance past a failing test (per `CLAUDE.md` line 18).

### 1. Project + Prisma scaffold
- **Test**: `tests/db/connection.test.ts` — instantiate the Prisma client and call `prisma.$connect()`; assert it resolves.
- **Implement**:
  - `npm init`, install `prisma`, `@prisma/client`, `vitest`, `tsx`, `typescript`.
  - `npx prisma init` (sets up `prisma/schema.prisma` + `.env` with `DATABASE_URL`).
  - `src/lib/prisma.ts` exporting a singleton client (avoid Prisma "too many connections" in tests via globalThis cache).
- **Commit**: `Scaffold backend project and Prisma client`.

### 2. Schema definition
- **Test**: a schema-validation test that imports the generated Prisma types and asserts the expected models and enums exist (TypeScript compile = green is the assertion).
- **Implement**: `prisma/schema.prisma` with:
  - `User` (id cuid, email unique, name, avatarUrl?, passwordHash? [null for OAuth users], provider enum, createdAt).
  - `MenuItem` (id cuid, name, description, price Decimal(10,2), category enum, tags String[], imageUrl, available Boolean default true).
  - `Order` (id cuid, userId fk, status enum, totalPrice Decimal(10,2), createdAt). Relation: `items OrderItem[]`.
  - `OrderItem` (id cuid, orderId fk cascade, menuItemId fk, quantity Int, unitPrice Decimal(10,2)).
  - `Conversation` (id cuid, userId fk nullable, sessionId String @unique, createdAt). Relation: `messages Message[]`.
  - `Message` (id cuid, conversationId fk cascade, role enum, content Json — Json keeps room for tool_use/tool_result blocks alongside plain text, createdAt).
  - Enums: `Provider`, `OrderStatus`, `MessageRole`, `Category`.
- Run `npx prisma validate` as part of `npm test` (or a precommit step).
- **Commit**: `Add Prisma schema with users, menu, orders, and conversations`.

### 3. Initial migration
- **Test**: post-migration, run a query to insert and read back a `User`; assert it round-trips.
- **Implement**: `npx prisma migrate dev --name init`. Commit the generated migration files.
- **Commit**: `Add initial database migration`.

### 4. Seed integration test
- **Test**: `tests/db/seed.test.ts` (runs seed against a test database, then asserts):
  - `prisma.menuItem.count()` ≥ 24.
  - Per-category counts: starters ≥ 6, mains ≥ 8, desserts ≥ 4, drinks ≥ 6.
  - At least one item carries each of the tags `vegan`, `vegetarian`, `spicy`, `gluten-free`, `signature`.
  - Every item has non-empty `description`, positive `price`, and an `imageUrl` starting with `https://`.
- **Commit**: `Add seed integration test`.

### 5. Seed implementation
- **Implement**: `prisma/seed.ts` with the 24+ curated items from `CLAUDE.md` lines 206–212. Each item: `name`, 1–2 sentence `description`, `price`, `category`, `tags`, Unsplash URL. Add `prisma: { seed: "tsx prisma/seed.ts" }` to `package.json`.
- Validate via `npx prisma db seed` locally, then re-run the seed test.
- **Commit**: `Seed menu with 24 bistro items`.

### 6. Test infrastructure
- **Implement**: a test setup script that creates an isolated test database (template: `bistro_test`) and runs `prisma migrate deploy` before each test run. Add `DATABASE_URL=...` for tests via `.env.test`.
- **Commit**: `Wire test database setup for Prisma tests`.

## Open questions

- Should `Conversation.userId` be required once auth is in place, or stay nullable to allow anonymous chat sessions? **Recommendation**: nullable; backfill on order confirmation if needed. Revisit once `/api/chat` is integrated with the frontend session model.
- Storage of `Message.content` as `Json` vs. flat `String`: keeping `Json` lets us persist tool_use/tool_result blocks exactly as Anthropic sends them, which is needed to replay history into the model on the next turn. This is the right call for the AI module — confirm during `ai-agent.md` implementation.
